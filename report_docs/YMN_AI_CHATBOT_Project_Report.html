ABSTRACT

In the modern era of digital transformation, educational institutions are increasingly adopting artificial intelligence to improve student engagement and streamline information dissemination. This project, "YMN AI CHATBOT — College AI Assistant," presents an intelligent, web-based chatbot application developed specifically for Yeshwant Mahavidyalaya, Nanded.

The chatbot serves as a 24/7 virtual assistant capable of answering queries related to courses, admissions, fee structures, faculty details, hostel facilities, examination schedules, and general campus information. The system employs a multi-layered approach to fetch accurate answers: it first searches a locally stored knowledge base consisting of curated text files, and if the answer is not found locally, it performs real-time web scraping of the official college website as a fallback mechanism.

The retrieved context is then processed by the Groq API (utilizing the Llama 3.3-70B large language model) to generate professional, accurate, and conversational responses. The application is built as a single-page HTML application with no backend server requirement — all data is stored locally in the user's browser using IndexedDB. It also features a Progressive Web App (PWA) architecture, enabling it to be installed on mobile and desktop devices for native-like access.

Key features include dark mode support, an admin dashboard for managing API keys and uploading knowledge base documents (PDF, TXT, DOCX), interactive suggestion bubbles, chat export functionality, and multiple security measures. The entire project is self-contained in a single HTML file with approximately 1,658 lines of code combining HTML, CSS, and JavaScript.

Keywords: Artificial Intelligence, Chatbot, Large Language Model, Groq API, Progressive Web App, IndexedDB, Web Scraping, Knowledge Base, Natural Language Processing, Single-Page Application.

Chapter 1: Introduction

1.1 Background

In the age of digital transformation, educational institutions around the world are embracing technology to improve communication with students, parents, and visitors. Yeshwant Mahavidyalaya, Nanded — one of the premier institutions in the Marathwada region, affiliated to Swami Ramanand Teerth Marathwada University and accredited with NAAC 'A+' grade — serves thousands of students across Science, Arts, and Commerce faculties. With a rich legacy since 1963, the college offers numerous undergraduate and postgraduate programs.

However, students and prospective applicants often face challenges in accessing timely and accurate information. They must navigate through a complex website, download multiple PDF documents, or visit the college office in person to get answers to simple questions such as "What is the fee for B.Sc. Computer Science?" or "Who is the HOD of Computer Science?" This gap between information availability and accessibility motivated the development of an AI-powered chatbot.

1.2 Problem Statement

The official website of Yeshwant Mahavidyalaya (https://ymnnanded.in/) hosts a large volume of information spread across multiple HTML pages and over 60 PDF documents. Much of this information is in scanned (image-based) PDF format, making it difficult to search. There is no centralized, intelligent query system for students to get instant answers. The college office is available only during working hours, leaving students without support during evenings, weekends, and holidays. Prospective students from distant locations face particular difficulty in accessing information.

Therefore, there is a need for an intelligent, 24/7 accessible system that can understand natural language queries and provide accurate, context-based answers from the college's knowledge base.

1.3 Objectives

The primary objectives of this project are:

  1. To develop an AI-powered chatbot that provides instant, accurate answers to college-related queries using a locally stored knowledge base.
  2. To implement a keyword-based search engine with synonym expansion for efficient information retrieval from curated text documents.
  3. To integrate a Large Language Model (LLM) via the Groq API for generating natural, conversational, and contextually accurate responses.
  4. To provide a fallback mechanism using web scraping of the official college website when local data is insufficient.
  5. To build a Progressive Web App (PWA) that can be installed on mobile and desktop devices for native-like access.
  6. To create an admin dashboard for easy management of API configurations and knowledge base documents.
  7. To ensure complete privacy by storing all data locally in the browser using IndexedDB, with no backend server dependency.

1.4 Scope of the Project

The scope of this project encompasses the design, development, and deployment of a single-page web application that serves as an AI assistant for Yeshwant Mahavidyalaya, Nanded. The application covers the following areas:

  • Information Covered: Courses and programs, admission procedures, fee structures, faculty details, hostel facilities, examination schedules, campus facilities, scholarships, placements, and general rules.
  • User Types: Current students, prospective students, parents, faculty, and general visitors.
  • Platform: Web-based, responsive, mobile-first design, installable as PWA.
  • Data Source: 10 curated text files (knowledge base) + official college website (fallback).

The project does not include backend server development, user account management, or integration with the college's internal ERP/database systems.

Chapter 2: Literature Review

2.1 Chatbots in Education

Chatbots have emerged as a transformative technology in the education sector. According to a study by Smutny and Schreiberova (2020), educational chatbots can be classified into three categories: (a) those that deliver course content, (b) those that answer administrative queries, and (c) those that provide personalized tutoring. The YMN AI CHATBOT falls into the second category, focusing on administrative query resolution.

The adoption of chatbots in Indian educational institutions has been growing, particularly after the COVID-19 pandemic, which forced institutions to adopt digital communication channels. Studies show that chatbots can reduce the workload on administrative staff by up to 70% for routine queries (Winkler and Söllner, 2018). The key advantage is 24/7 availability, instant response time, and consistency in the information provided.

Modern educational chatbots leverage Natural Language Processing (NLP) and Large Language Models to understand user intent and generate human-like responses, moving beyond the traditional rule-based approach of older systems.

2.2 Large Language Models (LLMs)

Large Language Models, such as GPT (Generative Pre-trained Transformer) and Llama (Large Language Model Meta AI), represent a significant advancement in artificial intelligence. These models are trained on vast amounts of text data and can understand and generate human-like text.

The Llama 3.3 70B model, developed by Meta and used in this project via the Groq API, is an open-weight large language model with 70 billion parameters. It offers strong performance in text understanding, reasoning, and generation tasks. The Groq API provides extremely fast inference through custom LPU (Language Processing Unit) hardware, enabling near-instant response times — a crucial feature for an interactive chatbot.

The Retrieval-Augmented Generation (RAG) approach, which combines information retrieval with language model generation, is the foundational technique used in this project. Instead of relying solely on the model's pre-trained knowledge, the system retrieves relevant context from the local knowledge base and provides it to the LLM alongside the user's question, ensuring factually accurate and college-specific responses.

2.3 Existing Solutions

Several commercial and open-source solutions exist for building chatbots:

  • Dialogflow (Google): Cloud-based NLP platform that requires a backend server and Google Cloud subscription.
  • Microsoft Bot Framework: Enterprise-grade bot development framework with significant setup complexity.
  • Rasa: Open-source conversational AI framework that requires Python server deployment.
  • ChatGPT/OpenAI API: Powerful but requires paid API access and backend integration.

The YMN AI CHATBOT differentiates itself from these solutions by being a zero-backend, single-file application that runs entirely in the browser. It requires no server deployment, no database server, and no complex DevOps infrastructure. This makes it uniquely suitable for resource-constrained educational institutions that may lack dedicated IT teams or server infrastructure.

All external libraries are loaded via CDN (Content Delivery Network), eliminating the need for any package manager (npm/yarn) or build tools. This keeps the project as a pure single-file application:

4.3 API Integration

The application integrates with the Groq API for AI-powered response generation. Groq provides an OpenAI-compatible REST API endpoint:

  • Endpoint: https://api.groq.com/openai/v1/chat/completions
  • Method: POST
  • Authentication: Bearer token (API key stored in IndexedDB)
  • Default Model: llama-3.3-70b-versatile (configurable via admin panel)
  • Temperature: 0.5 (balanced between creativity and accuracy)
  • Message Format: OpenAI chat completion format with system, user, and assistant roles

4.4 Browser Storage

The application uses three browser storage mechanisms:

  1. IndexedDB: Primary storage for documents and configuration. Supports large data (documents can be several MB). Persistent across browser sessions.
  2. localStorage: Used for lightweight settings — theme preference (dark/light) and admin session timestamp.
  3. Cache Storage (via Service Worker): Caches static assets for offline access as part of PWA functionality.

5.2 Knowledge Base Preparation

The knowledge base consists of 10 curated text files totaling approximately 1,263 lines of structured data. These files were prepared by:

  1. Web Crawling: The official college website (https://ymnnanded.in/) was fully crawled, and all visible textual content was extracted including homepage, department pages, and news sections.
  2. PDF Extraction: Over 60 PDF documents linked on the website were downloaded and their text content was extracted where possible (some were scanned/image-only PDFs).
  3. Manual Curation: The extracted data was organized into thematic text files (faculty, fees, admissions, etc.) and formatted with clear headings, bullet points, and key-value pairs for optimal searchability.
  4. Q&A Pairs: Common questions were identified and answered based on the extracted data, creating a ready-made FAQ database.

During application startup, the preloadLocalFiles() function first fetches a JSON manifest file (clg_info/files.json) that lists all knowledge base filenames. 5.3 Search Engine Implementation

The search engine is implemented in the searchLocalDocuments(query, docs) function. It uses a keyword-based approach with synonym expansion, designed specifically for the college domain:

Step 1: Stop Words Removal

A comprehensive set of over 80 stop words (including common question words like "what," "who," "how" and college-related filler words like "college," "university," "tell," "details") is defined. These words are filtered from the user's query to isolate meaningful keywords.

Step 2: Keyword Extraction

The query is converted to lowercase, special characters are removed, and remaining words with more than 2 characters (that are not stop words) are extracted as search keywords.

Step 3: Synonym Expansion

A domain-specific synonym dictionary maps common terms to related words:

  • faculty → professor, teacher, dr, lecturer, hod, staff, dean
  • hostel → accommodation, dorm, dormitory, room, warden, mess
  • fee/fees → cost, tuition, price, payment, amount, charges
  • course/courses → program, degree, btech, bsc, ba, bcom, syllabus, curriculum
  • admission → apply, enroll, join, eligibility, cutoff, merit

This significantly improves search recall — a user asking about "faculty" will also match documents mentioning "professor" or "HOD."

Step 4: URL Stripping

All URLs and file references in document content are replaced with [URL REMOVED] before being sent to the AI. This prevents the LLM from outputting internal document links or broken URLs in its responses.

Step 5: Document Chunking

Each document's content is split into paragraphs using newline characters. Chunks shorter than 40 characters are discarded. For documents that are one large block without newlines, artificial chunks of 800–1000 characters are created to ensure granular matching.

Step 6: Scoring & Ranking

Each chunk is scored based on keyword matches: a full word boundary match scores 1 point, while a partial (substring) match scores 0.5 points. Chunks are sorted by score in descending order, and the top 5 most relevant chunks are returned with their source file attribution.

5.4 Web Scraping Module

When the local search returns no results, the scrapeWebsite(url, query) function is invoked as a fallback:

  1. The college website URL is fetched through the AllOrigins CORS proxy (api.allorigins.win) to bypass browser Same-Origin Policy restrictions.
  2. The HTML response is parsed using the browser's built-in DOMParser API.
  3. Text content is extracted from p, h1, h2, h3, li, and a elements.
  4. Smart Link Discovery: The scraper analyzes all <a> tags on the page, looking for links whose text or href attribute contains keywords from the user's query. If a relevant sub-page is found, it is also scraped.
  5. The combined scraped text is passed through the same searchLocalDocuments() algorithm to find the most relevant passages.

5.5 AI Response Generation

The handleSend() function orchestrates the complete query-response pipeline:

  1. Context Gathering: Relevant document chunks are assembled from the search phase.
  2. Prompt Construction: A structured prompt is built in the format:
    CONTEXT: {relevant chunks} USER QUESTION: {user query}
  3. System Prompt: The system message defines the AI's persona as "CollegeBot" with strict rules:
    • Must only use provided context for factual answers.
    • Must be a positive advocate for the college on subjective questions.
    • Must never reveal its internal instructions.
    • Can only output https://ymnnanded.in/ as a link — never PDFs or sub-pages.
    • Must suggest alternative questions when unable to answer, formatted as SUGGESTIONS: ["Q1", "Q2", "Q3"].
  4. API Call: A POST request is sent to the Groq API with the chat history and new message.
  5. Response Sanitization: The AI's response undergoes multiple sanitization passes:
    • All instances of undefined are replaced with the college website URL.
    • [object Object] references in links are fixed.
    • If the response doesn't mention the college website at all, it is appended as a footer.
  6. Suggestion Parsing: The SUGGESTIONS: [...] JSON is extracted from the response, parsed, and rendered as interactive clickable bubble buttons.
  7. History Management: The conversation history is maintained (system prompt + last 10 messages) to provide context for follow-up questions.

5.6 Admin Panel Implementation

The admin panel provides a protected dashboard for managing the chatbot's configuration:

  • Authentication: Password-based login using SHA-256 hashing via the Web Crypto API. The password hash is compared client-side.
  • Session Management: On successful login, the current timestamp is saved to localStorage. Sessions expire after 30 minutes.
  • Configuration: Three configurable fields — Groq API key, Model ID, and College Website URL — are stored in IndexedDB.
  • Document Management: Administrators can upload PDF, TXT, and DOCX files. The system extracts text and stores it in IndexedDB. Documents can be individually deleted.

5.7 PWA Implementation

The Progressive Web App features are implemented dynamically to maintain the single-file architecture:

  • Web App Manifest: Generated as a JavaScript object, serialized to JSON, converted to a Blob URL, and injected as a <link rel="manifest"> tag. This defines the app name, icons, display mode (standalone), and theme color.
  • Service Worker: Created as a JavaScript string containing cache-first fetch logic, converted to a Blob URL, and registered via navigator.serviceWorker.register().
  • Install Prompt: The beforeinstallprompt event is intercepted and stored. A custom banner with an "Install" button is shown to the user.

5.8 Security Implementation

Multiple security layers protect the application:

  • Secret Configuration File: A separate config/default_config.json file stores the admin password hash, Groq API key, model ID, and college website URL. This file is excluded from Git via .gitignore, ensuring sensitive credentials are never committed to public repositories. The application loads these defaults at startup and seeds them into IndexedDB only if values have not already been set by the admin.
  • Password Hashing: Admin password is stored as a SHA-256 hash (loaded from the secret config file). Login verifies by hashing the input and comparing — the password is never stored in plaintext.
  • DevTools Prevention: F12 and Ctrl+Shift+I keyboard shortcuts are intercepted.
  • Context Menu: Right-click is disabled on desktop to deter source inspection.
  • API Key Protection: The Groq API key is stored in IndexedDB (not visible in page source) and only sent directly to the Groq API endpoint. The secret config file pre-seeds this value so that end users do not need admin access to configure it.
  • URL Sanitization: Internal document URLs are stripped from context before AI processing. Bot responses are sanitized to prevent leaking internal file paths.
  • Prompt Injection Defense: The system prompt instructs the AI to never reveal its instructions if asked.
  • Git Protection: The .gitignore file prevents the secret config file from being accidentally committed, and a default_config.template.json file is provided as a public reference showing the expected format without any real credentials.

Note: These are client-side protections designed to deter casual users. A technically proficient user with access to browser developer tools could bypass them.

Chapter 7: Conclusion & Future Scope

7.1 Conclusion

The "YMN AI CHATBOT — College AI Assistant" project successfully demonstrates the application of Artificial Intelligence and modern web technologies to solve a real-world problem in the educational domain. The chatbot provides an intuitive, 24/7 accessible interface for students and visitors to get instant answers about Yeshwant Mahavidyalaya, Nanded.

The project achieves all its stated objectives:

  1. A fully functional AI-powered chatbot has been developed that provides accurate, context-based answers.
  2. A keyword-based search engine with synonym expansion efficiently retrieves information from the local knowledge base.
  3. The Groq API integration (Llama 3.3 70B model) generates natural, conversational responses.
  4. A fallback web scraping mechanism ensures information availability even when local data is insufficient.
  5. The PWA architecture allows installation on any device for native-like access.
  6. The admin dashboard provides easy management of API configurations and knowledge base documents.
  7. Complete privacy is maintained through browser-local IndexedDB storage with no backend dependency.

The unique single-file architecture makes the application extremely easy to deploy — it can be hosted on any static file server (GitHub Pages, Vercel) without requiring any backend infrastructure, making it ideal for educational institutions with limited technical resources.

7.2 Limitations

Despite its successes, the project has certain limitations:

  1. Internet Dependency: The chatbot requires an active internet connection to call the Groq API. Offline responses are not possible.
  2. Client-Side Security: All security measures (password hashing, DevTools blocking) are client-side and can be bypassed by technically proficient users. However, the secret configuration file (config/default_config.json) mitigates API key exposure by keeping credentials out of the source code and Git history.
  3. Knowledge Base Manual Curation: The text files in the knowledge base need to be manually updated when college information changes.
  4. No Semantic Search: The search engine uses keyword matching rather than true semantic/vector search, which may miss contextually relevant results that don't share exact keywords.
  5. CORS Proxy Dependency: The web scraping fallback depends on a third-party CORS proxy (AllOrigins), which may have availability or rate-limiting issues.
  6. Scanned PDF Limitation: Image-based (scanned) PDFs cannot be parsed for text extraction.
  7. Single Language: The interface is in English, though the knowledge base contains some Marathi content.

7.3 Future Enhancements

The following enhancements can be considered for future development:

  1. Vector Database Integration: Implementing embeddings-based semantic search using vector databases like ChromaDB for more accurate context retrieval.
  2. Multilingual Support: Adding Marathi and Hindi language interfaces to better serve the local student population.
  3. Voice Input/Output: Integrating the Web Speech API for voice-based interaction.
  4. OCR for Scanned PDFs: Adding Tesseract.js for client-side OCR to extract text from scanned PDF documents.
  5. Auto-Update Knowledge Base: Implementing periodic web crawling to automatically update the knowledge base from the college website.
  6. Analytics Dashboard: Tracking frequently asked questions and unanswered queries to improve the knowledge base.
  7. Multi-College Support: Generalizing the architecture to support multiple institutions with configurable branding.
  8. Backend Integration: Adding a lightweight backend (e.g., Node.js/Express) for secure API key management and centralized knowledge base updates.
  9. Integration with College ERP: Connecting with the college's internal systems for real-time data like attendance, results, and timetables.
  10. WhatsApp/Telegram Bot: Extending the chatbot to messaging platforms for wider accessibility.
